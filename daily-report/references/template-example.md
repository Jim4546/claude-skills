# 日报 DocxXML 模板范例

> **受众**：同事 / 上级（对外）。不是 Claude 给用户的工作总结。
> **子任务三要素**：背景 / 核心动作（或核心决策）/ 改动内容（或产出）。
> **口吻**：第三人称客观陈述，禁第一人称，禁"Claude 帮我"。
> **风格基准**：本模板对齐历史日报真实文风——精简、给领导扫读，不是开发复盘文档。

## 完整骨架（最小可用范例）

```xml

<h3>1. v3.0d 官方公告源接入 + 来源可信度标签（commit <code>5800d6d</code>，已push并部署上线）</h3>
<ul>

  </li>
  <li><b>核心决策</b>：「权威等级」做成数据源的一个字段，后续再接新源只需填一个等级值，前端展示和模型优先级自动跟上</li>

<h2>五、收获与判断（技术学习）</h2>
<p>本板块面向刚入门编程的同事，把今天踩到 / 用到的技术概念拆开讲明白，便于从实践中积累。</p>
<ul>
  <li><b>同步 vs 异步（Synchronous / Asynchronous）</b>：同步 = 做完一件事才能做下一件，中途必须干等；异步 = 先把活交给后台，自己立刻去忙别的。v3.0e 上传改造就是把"干等几十秒"改成"1 秒返回、后台慢慢解析"。<b>记忆点</b>：用户体验卡不卡，常常就差一个"要不要让他干等"。</li>
  <li><b>数据库索引（Index）</b>：好比书的目录。一张表几万行，没索引找数据得一行行翻；建了索引数据库直接跳到目标位置。今天给文档表的「状态」字段加了索引。<b>记忆点</b>：经常做查询条件的字段值得加索引，但索引占空间、拖慢写入，不是越多越好。</li>
  <li><b>git 暂存区（Staging Area）</b>：提交代码分两步——先 git add 进暂存区，再 git commit。暂存区是全局唯一的，今天两个窗口并行就发生了"提交抢跑"。<b>记忆点</b>：提交完立刻用 <code>git show HEAD --stat</code> 核对实际提交了哪些文件。</li>
  <li><b>字符编码与乱码（UTF-8 / GBK）</b>：计算机存文字靠"编码"把字转成数字。同一份中文用 UTF-8 存、用 GBK 读就会乱码。<b>记忆点</b>：遇到乱码，先怀疑"存和读的编码对不上"。</li>
</ul>
```

## 必须遵守的格式约束

1. 第一行必须是 `<title>日报 · YYYY-MM-DD</title>`
2. 项目用 `<h2>{中文序号}、{项目名}项目进展</h2>`（一、二、三…），紧跟一句 `<p>` 导语
3. 子任务用 `<h3>{阿拉伯数字}. {标题}（commit <code>xxx</code>，已push）</h3>`——commit hash 写进标题尾括号
4. **子任务三要素固定**：`<b>背景</b>` / `<b>核心动作</b>`（或 `<b>核心决策</b>`，可并存）/ `<b>改动内容</b>`（或 `<b>产出</b>`）
5. 不在子任务里写「下一步」——后续统一收到「{项目}项目总结产出」末尾一条「待办：…」
6. 每个项目末尾必有 `<h3>{项目名}项目总结产出</h3>`，4-6 条要点，最后一条是「待办：…」
7. h2 之间、最后一个项目与末节之间、各末节之间都用 `<hr/>` 分隔
8. 末节固定三节且顺序固定：「今日总产出清单」→「整体待办事项」→「收获与判断（技术学习）」
9. 待办表 4 列：优先级 / 事项 / 责任人 / 截止时间
10. 「收获与判断（技术学习）」挑 6-9 个技术概念，每条 `<b>术语中文（术语英文）</b>` + 白话解释 + 今天用到处 + `<b>记忆点</b>`
11. 正文末尾留一个空 `<p></p>`

## 口吻硬约束

- 全文**禁用第一人称**（我 / 我们 / 咱们）
- **禁止 AI/对话词**：Claude 帮、AI 协作、用户提到、用户表态、经讨论、对话中
- 主语用项目 / 模块 / 工作本身
- 口语词清理：OK、那么、其实、是不是、有点

## 措辞约束（受众是非计算机专业、但有些技术背景的同事 / 上级）

- 代码标识符 / 文件路径 / 函数名 / 常量必须翻译成业务语言（详见 publish.md §4「受众与措辞硬约束」）
- 子任务 h3 标题尤其要让非程序员看懂"做了什么、解决了什么问题"
- **例外**：「收获与判断（技术学习）」板块本就是教学，可以正常用术语，但要逐个解释

## 篇幅与精简

- 一个项目控制在 3-5 个子任务，过细的合并
- 每个子任务核心动作 3-5 条 bullet
- 不写大段方法论反思——这是给领导扫读的汇报。深度技术沉淀放到「收获与判断（技术学习）」板块，且面向新手、控制在每条 2-3 句

## 单项目场景

- 仍用 `<h2>一、{项目名}项目进展</h2>`（不省略序号）
- 仍写「{项目名}项目总结产出」
- 末节顺接为「二、今日总产出清单」「三、整体待办事项」「四、收获与判断（技术学习）」
