## 格式约束（硬）

1. 第一行 `<title>王佳玮工作日报_{yyyymmdd}</title>`，`{yyyymmdd}` = date 去掉横线的 8 位（`2026-06-12`→`20260612`）。此 `<title>` 必须等于发布用的 `$title`（建节点/查重/推群同一来源，见 publish.md）。
2. `<title>` 之后、每个 `<h2>` 之间都用 `<hr/>` 分隔。
3. 项目用 `<h2>{中文序号}、{项目名}</h2>`（一、二、三…），按工作量倒序，最大的放最前。
4. 项目下用若干 `<p>✅ {子主题}</p>`，每个子主题后跟 0-1 个 `<ul>`，**2-4 条**精简要点（参考文档里有的子主题不带 `<ul>`，一句话写进 `<p>` 也可）。
5. 每个项目末尾**必须**有 `<h3>相关文档</h3>` + `<ul>`（该项目涉及的飞书链接；该项目确实没有任何飞书链接时，此节可省略）。
6. 末尾兜底节 `<h2>{中文序号}、其他：沟通与会议</h2>`：收纳映射不到任何项目的飞书纯沟通结论 + 杂会（会议名 时间 一句结论 + 妙记链接）。没有则省略整节。

## 不要的东西（和旧详细日报的区别）

- ❌ 不写子任务三要素（背景/核心动作/改动内容）。
- ❌ 不写「收获与判断（技术学习）」板块。
- ❌ 不写整体待办事项表格。
- ❌ 不写 worklog 的个人视角（否决 AI / 原话证据 / 工作逻辑流程图）。
- 一句话：**只要 `✅ 主题 + 几条要点 + 链接`**。

## 链接装配（每项目「相关文档」）

扫该项目所有来源（本地对话 + 飞书会话 + 会议）里的飞书 URL：
- 正则：`https?://[^\s)]*feishu\.cn/(base|docx|wiki|sheets|file)/[A-Za-z0-9]+(\?[^\s)]*)?`
- 去重后按出现顺序保留**前 5** 条。
- **链接文字一律用文档原始标题，不要自己编名/标类型后缀**：对每条 URL 调
  `lark-cli drive +inspect --url '<url>' --as user --format json`，取返回的 `title` 字段作为 `<a>` 文字（wiki 链接 inspect 会自动 unwrap 到底层文档，返回其真实标题）。
  inspect 失败 / 无权限 / 取不到 title → 兜底用裸 URL。
- 妙记链接 `https://<host>/minutes/<token>` 放在对应会议那条里，不进「相关文档」。

## 业务语言翻译（受众非程序员）

代码标识符 / 文件路径 / 函数名 / 常量必须翻译成业务语言再装入：
- `setImmediate` 兜底 → 「上传后立即返回、后台慢慢解析」
- `extractText` → 「提取正文」
- `src/app/api/documents/route.ts` → 「上传 API」
- `prisma/schema.prisma` → 「数据库结构定义文件」
- `tier=official` → 「权威等级标为官方」
- 缩写首次出现要解释：SPA → 「用 JS 动态渲染的现代网页」；413 → 「请求体过大」
- **必须保留**：commit hash、版本号（v3.0d）、生产 IP/域名、字段数/数据源数等可核对数据、第三方系统名（港交所、Caddy、pm2）。

判断标准：把每个 `✅ 主题` 和要点读一遍——非程序员能不能看懂「做了什么、解决了什么」？看不懂就改。

## 口吻硬约束

- 全文**禁用第一人称**（我 / 我们 / 咱们）。
- **禁止 AI/对话词**：Claude 帮、AI 协作、用户提到、用户表态、经讨论、对话中。
- 主语用项目 / 模块 / 工作本身；用「完成 X」「新增 Y」「确认 Z」客观陈述。
- 口语词清理：OK、那么、其实、是不是、有点。
- 不凭空捏造，没做成/没证据的不写。

## 篇幅

- 一个项目 3-6 个 `✅ 子主题`，过细的合并。
- 每个子主题 2-4 条要点，不写大段反思。
